# 발송 신뢰성 설계 — Outbox Pattern + Batch 처리

---

## 1. 왜 필요한가 — 현재 시스템의 문제

현재 메시지 발송 흐름은 다음과 같습니다.

```
사용자 요청 → 업체 API 직접 호출 → 성공하면 DB 저장
```

이 구조에서 아래 상황이 발생하면 **메시지가 조용히 사라집니다.**

| 시나리오 | 결과 |
|----------|------|
| 업체 API 서버가 일시적으로 다운됨 | 발송 실패, 재시도 없음 |
| 네트워크 타임아웃 발생 | 발송됐는지 안 됐는지 불명확 |
| API 호출 도중 서버 재시작 | 진행 중이던 발송 유실 |

**핵심 문제**: 메시지를 보내겠다는 의도는 있었지만, 실제로 전달됐는지 보장할 방법이 없습니다.

---

## 2. Outbox Pattern — "일단 적어두고, 나중에 보낸다"

### 비유

> 은행 송금을 예로 들면, 창구 직원은 고객의 송금 요청을 **즉시 타 은행으로 전달하지 않습니다.**
> 먼저 내부 장부에 "이 금액을 보내야 한다"고 기록하고, 별도의 정산 프로세스가 실제 이체를 처리합니다.
> 중간에 시스템 장애가 나도 장부에 기록이 남아있어 반드시 처리됩니다.

메시지 발송도 동일한 원리를 적용합니다.

### 변경 후 흐름

```
[1단계] 사용자 요청 수신
         ↓
    "보내야 할 메시지" 목록(outbox)에 기록  ← 이것만 보장하면 됨
    → 사용자에게 즉시 응답 반환

[2단계] 별도 처리기(Batch)가 주기적으로 실행
         ↓
    outbox 목록에서 "아직 안 보낸 것" 확인
         ↓
    업체 API 호출
         ↓
    성공 → "보냈음"으로 표시
    실패 → 재시도 예약
```

### 상태 흐름

```
PENDING  →  (Batch 처리 중)  →  SENT
                                FAILED  (재시도 횟수 초과)
```

| 상태 | 의미 |
|------|------|
| PENDING | 발송 대기 중. 아직 업체 API에 전달 안 됨 |
| SENT | 업체 API 호출 성공. 발송 완료 |
| FAILED | 최대 재시도 횟수 초과. 수동 확인 필요 |

## 3. Batch 처리 — "묶어서 효율적으로"

### [설계 결정] 단순 스케줄러 vs Spring Batch

| 항목 | 단순 `@Scheduled` + 루프 | Spring Batch |
|------|--------------------------|--------------|
| 청크 처리 | 직접 구현 필요 | Reader → Processor → Writer 청크 내장 |
| 재시도 정책 | 직접 구현 필요 | `RetryTemplate`, 지수 백오프 내장 |
| 잡 이력 관리 | 없음 | 실행 시각, 처리 건수, 실패 원인 자동 기록 |
| 실패 후 재시작 | 처음부터 재처리 | 실패 지점부터 재시작 가능 |

**선택 → Spring Batch**

재시도 정책과 청크 처리는 단순 스케줄러로도 구현할 수 있지만, 직접 작성한 코드는 엣지 케이스(서버 재시작, 부분 실패 등)에서 예측하기 어려운 동작이 발생할 수 있다. Spring Batch는 이러한 운영 시나리오가 이미 검증된 구현으로 내장되어 있어, 신뢰성 보장에 집중할 수 있다.

---

### 왜 묶어서 처리하나

발송 요청이 폭발적으로 들어오는 상황을 가정합니다.

> 오전 9시, 100개 조직이 동시에 각자의 고객에게 알림톡을 발송 요청.
> 총 50,000건의 발송 요청이 1분 안에 들어옴.

이를 건 단위로 실시간 처리하면 업체 API에 순간 부하가 집중됩니다. Batch는 이를 **청크(묶음) 단위**로 처리합니다.

### 청크 처리 예시

```
설정: 청크 크기 = 100건, 실행 주기 = 10초

[10초마다 실행]
  읽기:   PENDING 상태 100건 조회
  처리:   조직별 업체 분기 → API 호출
  쓰기:   100건 상태 일괄 업데이트
  반복
```

50,000건 기준으로 계산하면:

| 항목 | 값 |
|------|-----|
| 청크 크기 | 100건 |
| 실행 주기 | 10초 |
| 처리량 | 약 600건/분 |
| 전체 처리 예상 시간 | 약 83분 |

> 실제 업체 API 응답 속도와 서버 리소스에 따라 조정 가능한 값입니다.

## 4. 재시도 정책 — "포기하지 않되, 폭격하지 않는다"

### 문제 상황

업체 API가 일시적으로 응답하지 않을 때, 즉시 재시도를 반복하면 두 가지 문제가 생깁니다.

1. 이미 불안정한 업체 서버에 추가 부하를 줌 → 회복을 방해
2. 우리 서버 자원도 소진

### 지수 백오프 + Jitter 전략

재시도 간격을 **점점 늘리는** 방식입니다.

```
1차 실패 → 10초 후 재시도
2차 실패 → 20초 후 재시도  (간격 2배)
3차 실패 → 40초 후 재시도  (간격 2배)
4차 실패 → FAILED 처리 (더 이상 재시도 없음)
```

**Jitter(흔들림)**: 여러 메시지가 동시에 재시도하는 것을 방지하기 위해 각 재시도 간격에 약간의 랜덤 편차를 추가합니다.

```
예: 2차 재시도 대기 시간 = 20초 ± 최대 5초 랜덤
```

### 재시도 대상 / 비대상

| 상황 | 처리 |
|------|------|
| 업체 서버 오류 (5xx), 타임아웃 | 재시도 대상 — 일시적 장애로 판단 |
| 잘못된 요청 (4xx) | 재시도 불필요 — 요청 자체가 잘못됨 |
| 일일 발송 한도 초과 (Daily Limit) | 재시도 불필요 — 당일 추가 발송 불가. 즉시 FAILED 처리 후 클라이언트에 `DAILY_LIMIT_EXCEEDED` 응답 |

## 5. 장애 시나리오별 동작

### 시나리오 A: 업체 API 일시 다운

```
1. 발송 요청 수신 → outbox에 PENDING 기록 ✅
2. Batch 실행 → 업체 API 호출 실패 (타임아웃)
3. 10초 후 재시도 → 실패
4. 20초 후 재시도 → 성공 ✅ (업체 복구됨)
5. outbox 상태 → SENT
```

결과: 발송 지연은 발생했지만 **유실 없음**.

---

### 시나리오 B: 재시도 3회 모두 실패

```
1. 발송 요청 수신 → outbox에 PENDING 기록 ✅
2. Batch 실행 → 1차, 2차, 3차 모두 실패
3. outbox 상태 → FAILED
4. 운영팀 알림 발송 (모니터링 연동)
```

결과: 발송 실패. **outbox에 기록이 남아있어** 수동 재처리 가능.

---

### 시나리오 C: Batch 실행 도중 서버 재시작

```
1. PENDING 100건 읽기 완료
2. 50건 처리 중 서버 재시작
3. 서버 재시작 후 Batch 재실행
4. 처리 안 된 50건은 여전히 PENDING → 다시 처리
```

결과: **중복 발송 가능성**이 있습니다.

> Spring Batch의 청크 단위 커밋으로 이 위험을 최소화한다. 청크 단위로 상태가 SENT로 커밋되므로, 서버 재시작 후에는 미커밋 청크의 건만 재처리된다. 다만 업체 API 호출 성공 후 DB 커밋 전 서버가 죽는 순간에는 중복 발송이 발생할 수 있으며, 이는 현 단계에서 허용하는 트레이드오프다.

---

## 6. 발송 내역 기록 — 비동기 저장

Batch가 발송 완료 후 상세 내역(과금 정보, 상태 이력 등)을 DB에 저장할 때, **발송 처리와 분리하여 비동기로 처리**합니다.

```
Batch: 발송 완료 → outbox 상태 SENT 업데이트
                 ↓ (이벤트 발행)
별도 스레드:    message_logs 저장
                업체별 상세 로그 저장 (cloud_message_logs / sendtalk_logs)
```

**이유**: 로그 저장이 느려도 다음 발송 처리에 영향을 주지 않기 위함입니다.

**트레이드오프**: 비동기 저장 중 서버가 재시작되면 상세 로그가 유실될 수 있습니다. 발송 성공 여부는 outbox로 보장되므로, 현 단계에서는 허용 가능한 수준으로 판단합니다.
