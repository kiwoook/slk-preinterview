# 요구사항 분석

## 1. 현재 시스템 분석

### 도메인 모델

```
Organization (고객사)
  └── UserGroup (사용자 그룹, 1개 이상)
        └── User (사용자, 1개 이상의 그룹에 소속)
```

### 현재 메시지 발송 흐름

```
[발송 요청] → [MessageService] → [CloudMessageClient] → [클라우드메시지 API]
                                                               ↓
                                                       [MessageLog DB 저장]
```

- 앱 푸시를 **제외한** 모든 메시지(SMS, MMS, 알림톡)는 클라우드메시지 단일 업체를 통해 발송
- 발송 내역은 DB에 저장

## 3. 가정 사항 (사고 과정 포함)

내부 구현을 정확히 알 수 없으므로 각 가정의 **고민 → 선택 → 근거** 과정을 함께 기록합니다.

---

### A-01. 메시지 발송 API 범위

**선택** → 실제 메시지를 발송하는 API(`POST /messages` 등)는 기존 시스템에 이미 존재한다고 가정. 이번 변경 범위는 **조직별 업체 설정을 관리하는 API**에 한정된다.

**근거**
- 과제의 핵심은 "어떤 업체로 보낼지"를 조직별로 다르게 설정하는 것이다. 메시지 발송 자체의 흐름은 기존에 구현되어 있으며, 내부적으로 Provider 레이어를 거치도록 수정하는 방식으로 대응한다.
- 새로 추가되는 API는 업체 설정 조회(`GET`), 업체 변경(`PUT`), 연결 테스트(`POST /test`) 3개에 한정된다.

---

### A-02. 수신자 관리 범위

**선택** → 수신자(전화번호)는 발송 요청 호출자가 이미 확정하여 전달한다고 가정. 수신자 목록 관리 및 사용자 그룹 기능은 이 시스템의 범위 밖이다.

**근거**
- 이번 변경의 핵심은 조직별 업체 분기이며, 수신자 관리 레이어 추가는 별도 도메인 설계가 수반된다.
- 도메인 모델에 UserGroup, User 개념이 존재하나, 발송 시점에 수신자를 해소(resolve)하는 로직은 상위 레이어(호출자)가 담당한다고 가정한다.

---

### A-03. 앱 푸시 처리 방식

**선택** → 앱 푸시는 FCM/APNs 기반의 별도 시스템에서 처리되고 있다고 가정. 이번 변경 범위에서 제외.

**근거** → 앱 푸시는 전화번호 기반의 SMS/MMS/알림톡과 달리 디바이스 토큰 기반으로 FCM/APNs에 직접 연동되는 별개 채널이다. 이번 변경과 무관하다.

---

### A-04. Organization 테이블 존재 여부

**선택** → Organization 테이블이 존재하며 PK(id)를 가진다고 가정

**근거** → 과제 배경에서 Organization 도메인이 명시되어 있으므로 자명하다.

---

### A-05. 인증 및 권한 처리 방식

**고민**
- 조직 관리자만 업체 설정을 변경할 수 있어야 한다.
- 내부 인증 구현 방식을 알 수 없다.

**선택** → JWT Bearer 토큰 기반 인증이 존재하며, 토큰에 사용자 역할(role)과 소속 조직(orgId) 정보가 포함된다고 가정

**근거** → B2B SaaS에서 일반적으로 사용되는 인증 방식이다. 구체적인 구현은 내부 시스템에 의존하므로 가정으로만 기록한다.

```
API 접근 조건 (가정)
- Authorization: Bearer {token}
- token.role = ADMIN
- token.orgId = 요청 경로의 {orgId}
```

---

### A-06. MessageService 내부 구조

**고민**
- MessageService가 업체 클라이언트를 직접 호출하는지, 이미 추상화된 레이어가 있는지 알 수 없다.

**선택** → MessageService가 CloudMessage Client를 직접 호출하는 구조로 가정 (추상화 없음)

**근거**
- 단일 업체만 사용하는 상황에서 인터페이스 추상화를 미리 만들 동기가 없다.
- **설계 영향**: Strategy Pattern 적용을 위한 추상화 레이어를 신규 구현해야 하며, 기존 MessageService 호출부를 리팩토링해야 한다.

---

### A-07. 업체 변경 시 진행 중인 발송 처리

**고민**
- 조직 관리자가 업체를 변경하는 시점에, 이미 발송 요청 중인 메시지가 있을 수 있다.
- 기존 업체로 계속 처리해야 하는가, 아니면 취소해야 하는가?

**선택** → 진행 중인 발송은 기존 업체로 계속 처리한다. 업체 변경은 **다음 발송 요청부터** 적용된다.

**근거**
- 사용자는 메시지를 요청한 시점에 발송이 완료된다고 가정하고 업체를 변경할 수 있다. 변경으로 인해 진행 중인 발송이 취소되면 사용자 경험을 저해한다.
- 발송 취소가 필요한 경우는 별도의 취소 요청 흐름으로 처리해야 한다.
- 기술적으로도 이미 업체 API에 전달된 요청은 서버 측에서 중단할 수 없다.
- 업체 설정 변경은 DB 단일 레코드 업데이트(atomic)이므로, 그 이후 새로 들어오는 요청은 자동으로 새 업체를 사용한다. 별도 처리 불필요.

---

### A-08. 인증 정보(Credential) 관리 방식

> **[재설계]** 초기 가정(전역 환경변수)에서 변경됨.
>
> 재설계 배경: 각 Organization이 메시지 업체와 직접 계약하는 Integration 모델로 확정됨. 조직별 계약·과금 독립성 확보 및 보안 격리가 핵심 요인.

**고민**

A사가 업체와 직접 계약한 Reseller 모델인지, 각 조직이 자체 계약을 가진 Integration 모델인지에 따라 설계가 달라진다.

| 판단 기준 | Reseller (Model 1) | Integration (Model 2) |
|-----------|--------------------|-----------------------|
| 과금 주체 | A사가 업체에 지불 후 조직에 재청구 | 각 조직이 업체에 직접 지불 |
| 알림톡 채널 | A사 대표 채널 공유 | 각 조직 자체 채널 필수 |
| "화면에서 입력하는 값" | 발신번호, 프로필 ID 정도 | API 키 포함 전체 Credential |

**선택** → Integration 모델 (Model 2) — 각 조직이 자체 Credential을 보유, DB에 암호화 저장

**근거**
- **계약·과금 독립성**: 각 조직이 업체와 직접 계약하므로 과금 단위, 발송 한도, 요금제가 조직별로 독립된다. 전역 Credential로는 조직 간 과금 경계를 보장할 수 없다.
- **조직 단위 보안 격리**: 하나의 Credential을 공유하면 특정 조직의 오남용이 전체 서비스 발송에 영향을 줄 수 있다. 조직별 독립 Credential로 격리한다.
- "서비스 제공 업체에서 필요로 하는 값을 화면에서 입력할 수 있어야 한다"는 요구사항이 API 키를 포함한 전체 Credential 입력을 의미한다.

> 상세 저장 설계는 [ADR-002](../architecture/adr.md#adr-002-credential-및-조직-업체-매핑-저장-방식) 참고

### A-09. 조직-업체 매핑 및 Credential 저장 방식

> **[재설계]** A-08 재설계에 따라 함께 변경됨.

**선택** → 별도 테이블 `organization_message_providers` — provider_type + 암호화 JSON Credential을 함께 관리

**근거**
- Credential을 조직별로 저장해야 하므로 Organization 테이블 컬럼 추가로는 수용 불가능하다.
- 두 업체의 Credential 구조가 이질적이므로 JSON 컬럼이 가장 유연하다.
- Organization 테이블을 오염시키지 않기 위해 별도 테이블로 분리한다.

> 상세 스키마는 [ADR-002](../architecture/adr.md#adr-002-credential-및-조직-업체-매핑-저장-방식) 참고

### A-10. 발송 내역 테이블 설계

**고민**
- 기존 발송 내역 테이블은 단일 업체 시절 설계되어 어떤 업체로 발송됐는지 기록이 없을 것이다.
- 업체가 다변화되면 업체별 응답 데이터 구조가 달라 하나의 테이블로 수용하기 어렵다.

**선택** → 공통 기본 테이블(`message_logs`) + 업체별 상세 테이블 분리, `provider_type` 반정규화

**근거** → 상세 설계는 [ADR-003](../architecture/adr.md#adr-003-발송-내역messagelog-테이블-설계) 참고

### A-11. 클라우드메시지 알림톡 채널(senderProfileId) 수

**고민**
- 클라우드메시지는 알림톡 발송 요청 시 `sender.profileId`를 매번 지정하는 구조다.
- 이는 1개의 API Key로 여러 카카오 채널을 사용할 수 있음을 시사한다.
- 반면 센드톡은 API Key 발급 시 채널이 고정되어 별도 지정이 없다.
- 한 조직이 목적별로 여러 채널을 쓰는 경우(예: 공지용 채널 / 마케팅용 채널)를 지원해야 하는가?

**선택** → 조직당 1개의 `senderProfileId`만 저장. 다중 채널은 이번 과제 범위 밖으로 가정.

**근거**
- 과제 요구사항은 "한 조직은 하나의 업체만 사용"으로 명시되어 있으며, 채널 수에 대한 언급이 없다.
- B2B SaaS에서 조직별 단일 알림 채널이 일반적인 초기 설계다.
- 향후 다중 채널이 필요해지면 `organization_message_providers` 테이블의 JSON Credential에 배열 구조로 확장 가능하다. 현재 설계가 이를 막지 않는다.
