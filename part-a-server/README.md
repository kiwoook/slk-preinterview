# Part A. 서버 설계

**현재 상태:** 설계 완료

---

## PM을 위한 개요

### 무엇을, 왜 바꾸는가

> 지금은 모든 고객사가 하나의 메시지 발송 업체를 공유합니다.
> 고객사마다 계약한 업체가 다를 수 있는 상황에 대응하기 위해,
> **고객사별로 원하는 업체를 직접 설정하고 사용할 수 있도록** 시스템을 변경합니다.

### 변경 전 / 후

| | 변경 전 | 변경 후 |
|--|---------|---------|
| 업체 구조 | 전체 고객사가 클라우드메시지 단일 사용 | 고객사별 독립 업체 설정 |
| 업체 설정 | 서버 설정값으로 고정 | 관리자 화면에서 직접 입력·변경 |
| 지원 업체 | 클라우드메시지 | 클라우드메시지 + 센드톡 |
| 발송 실패 시 | 재시도 없음, 유실 가능 | 자동 재시도, 기록 보존 |

### 핵심 결정 요약

| 결정 | 선택한 방향 | 이유 |
|------|------------|------|
| 인증 정보 관리 | 고객사별 DB 저장 + 암호화 | 알림톡 채널은 사업자마다 별도 발급되므로 공유 불가 |
| 업체 전환 방식 | 다음 발송 요청부터 적용 | 진행 중인 발송을 취소하면 사용자 경험 저해 |
| 발송 신뢰성 | 요청을 먼저 기록 후 발송 처리 | 업체 서버 장애 시에도 메시지 유실 방지 |

### 리스크와 대응

| 위험 | 영향 | 대응 방법 |
|------|------|-----------|
| 기존 고객사 발송 중단 | 전체 고객사 | 마이그레이션 시 기존 업체(클라우드메시지)를 기본값으로 자동 유지 |
| 인증 정보 유출 | 고객사별 API 키 | 저장 시 암호화, 화면·로그에서 마스킹 처리 |
| 발송 한도 초과 | 당일 발송 불가 | 한도 초과 시 즉시 감지하여 관리자에게 안내 |

## 개발자를 위한 문서 가이드

### 권장 읽기 순서

```
1. 요구사항 + 가정 사항   → 변경 범위와 설계 전제 파악
2. 시스템 설계            → 전체 아키텍처와 DB 구조 파악
3. 기술적 의사결정 (ADR)  → 왜 이렇게 설계했는지 근거 확인
4. 세부 설계 문서         → 필요한 영역만 선택 참조
5. API 명세               → 엔드포인트 구현 참조
6. 검증 및 롤백           → 배포 전 체크리스트 확인
```
