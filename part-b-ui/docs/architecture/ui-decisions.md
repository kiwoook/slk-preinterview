# UI 결정 사항 (ADR)

설계 시 고민한 포인트와 선택 근거를 기록합니다.
형식: 고민 → 대안 비교 → 선택 → 근거

---

## UI-01. 현재 설정 표시 방식

**고민**
업체 변경 화면에서 운용자가 "현재 무엇이 적용 중인지" 모르면 실수할 가능성이 높다.
현재 상태를 어떻게, 어디에 보여줄 것인가?

| 대안 | 설명 |
|------|------|
| A. 편집 폼에 현재 값 채워두기 | 편집과 조회가 같은 공간. 현재 값인지 새 값인지 구분 어려움 |
| B. 별도 읽기 전용 카드로 분리 | 현재 상태와 편집 영역이 시각적으로 분리되어 명확 |
| C. 모달/사이드패널로 현재 설정 확인 | 추가 인터랙션 필요, UX 흐름 끊김 |

**선택** → B. 별도 읽기 전용 카드 (상단 고정)

**근거**
- 운용자는 값을 입력하면서도 "지금 뭐가 적용 중인지" 항상 볼 수 있어야 한다.
- 카드 배경색(`bg-slate-50`)으로 편집 영역과 시각적 구분을 주면 인지 부하가 낮다.
- 저장 성공 시 카드가 즉시 갱신되어 변경 결과를 바로 확인할 수 있다.

## UI-02. 업체 선택 UI 컴포넌트

**고민**
클라우드메시지 / 센드톡 중 하나를 선택하는 UI를 어떤 컴포넌트로 구현할 것인가?

| 대안 | 설명 |
|------|------|
| A. 라디오 버튼 | 선택지 나열, 현재 선택 상태 명확 |
| B. 드롭다운(Select) | 선택지가 숨겨짐, 현재 상태 한눈에 파악 어려움 |
| C. 카드형 선택 버튼 | 업체 로고·설명 포함 가능, 시각적으로 풍부하나 구현 비용 높음 |

**선택** → A. 라디오 버튼

**근거**
- 선택지가 2개로 고정이므로 드롭다운의 장점(공간 절약)이 없다.
- 현재 선택 상태가 항상 노출되어 실수를 줄인다.
- B2B 설정 화면에서 가독성과 단순함이 우선이다.

## UI-03. Credential 입력 필드 표시 방식

**고민**
기존에 저장된 Credential을 화면에서 어떻게 표시할 것인가?
보안(노출 방지)과 사용성(현재 값 확인) 사이의 트레이드오프가 있다.

| 대안 | 설명 |
|------|------|
| A. 전체 마스킹 (`••••••••`) | 보안은 좋으나 어떤 값이 저장됐는지 전혀 알 수 없음 |
| B. 앞 4자리 + 마스킹 (`AKCL••••••••`) | 어떤 키가 설정됐는지 힌트 제공, 실수 감지 가능 |
| C. 전체 노출 | 보안 취약, 화면 캡처/어깨너머 노출 위험 |

**선택** → B. 앞 4자리 + 마스킹 (서버에서 가공하여 반환)

**근거**
- 운용자가 "올바른 키가 저장되어 있는지" 힌트 수준에서 확인 가능하다.
- 완전한 키 노출 없이 식별 가능한 정보(앞 4자리)만 제공한다.
- 서버에서 마스킹 처리 후 반환하므로 클라이언트에 원본이 내려오지 않는다.

## UI-04. 연결 테스트 위치 및 필수 여부

**고민**
연결 테스트 버튼을 어디에 배치하고, 저장 전에 테스트를 강제할 것인가?

| 대안 | 설명 |
|------|------|
| A. 저장 전 테스트 강제 (테스트 성공해야 저장 활성화) | 잘못된 Credential 저장 원천 차단. 테스트 API 실패 시 저장 불가 |
| B. 테스트는 선택, 저장과 독립 | 유연하나 테스트 없이 잘못된 값 저장 가능 |
| C. 저장과 테스트를 통합 (저장 시 자동 테스트) | UX 단순화. 테스트 실패 시 저장 롤백 필요, 복잡도 증가 |

**선택** → A. 저장 전 테스트 강제 (테스트 실패 상태에서 [저장] 비활성화)

**근거**
- 잘못된 Credential이 저장되면 이후 모든 발송이 실패한다. 사전 차단이 최선이다.
- 단, 연결 테스트 자체가 우선순위 낮은 기능(API 미구현 가능성)이므로 테스트 미실행 상태는 저장 허용한다. 테스트를 했는데 실패한 경우에만 [저장] 비활성화.

## UI-05. 업체 변경 시 확인 다이얼로그

**고민**
업체 변경은 발송 전체에 영향을 준다. 운용자 실수를 어떻게 방지할 것인가?

| 대안 | 설명 |
|------|------|
| A. 확인 다이얼로그 표시 | 의도적 행위 재확인, 진행 중 발송 안내 가능 |
| B. 저장 즉시 적용, 취소 기능 제공 | 빠른 UX이나 취소 전 발송 오류 발생 가능 |
| C. 별도 확인 입력 (업체명 타이핑) | 과도한 마찰, B2B 설정 화면에 부적합 |

**선택** → A. 확인 다이얼로그 (업체 변경 시에만, Credential 변경은 생략)

**근거**
- 업체 변경은 비가역적 영향(다음 발송부터 즉시 적용)이므로 한 번의 재확인이 적절하다.
- 다이얼로그에 "진행 중인 발송은 기존 업체로 완료됩니다"를 명시해 운용자의 불안을 해소한다.
- 동일 업체에서 Credential만 바꾸는 경우는 영향 범위가 작아 다이얼로그 불필요.
