# Part B — 문자 발송 업체 설정 화면 설계

**현재 상태:** 설계 완료

---

## PM을 위한 개요

### 설계 목표

Part A에서 구현한 조직별 업체 설정 기능을 관리자가 직접 조작하는 화면입니다. 잘못된 Credential이 저장되면 해당 조직의 발송이 전면 중단되므로, **운용자 실수를 줄이는 것**을 최우선으로 설계했습니다.

- 현재 적용 중인 설정을 편집 중에도 항상 확인할 수 있어야 한다
- 잘못된 값은 저장 전에 차단한다 (연결 테스트)
- 업체 변경처럼 영향 범위가 큰 행위는 의도를 한 번 더 확인한다

### 핵심 결정 요약

| 결정 | 선택한 방향 | 이유 |
|------|------------|------|
| 현재 설정 표시 | 별도 읽기 전용 카드 (상단 고정) | 편집 중에도 현재 상태를 항상 확인 |
| 업체 선택 UI | 라디오 버튼 | 선택지 2개 고정, 현재 상태 항상 노출 |
| Credential 표시 | 앞 4자리 + 마스킹 | 식별 가능 수준 힌트, 원본 비노출 |
| 연결 테스트 | 실패 시 저장 비활성화 | 잘못된 Credential 저장 사전 차단 |
| 업체 변경 확인 | 확인 다이얼로그 (업체 변경 시에만) | 비가역적 영향 재확인, Credential 변경만은 생략 |

### 리스크와 대응

| 위험 | 영향 | 대응 방법 |
|------|------|-----------|
| 잘못된 Credential 저장 | 해당 조직 발송 전면 중단 | 저장 전 연결 테스트 강제. 테스트 실패 시 저장 버튼 비활성화 |
| 업체 변경 실수 | 다음 발송부터 업체 전환 | 확인 다이얼로그에 "다음 발송부터 적용" 명시 |
| 미저장 상태 이탈 | 입력값 손실 | 미저장 변경 시 페이지 이탈 차단 다이얼로그 |

## 개발자를 위한 문서 가이드

### 권장 읽기 순서

```
1. 화면 흐름       → 전체 사용자 흐름과 API 호출 시점 파악
2. UI 결정 사항    → 설계 고민 포인트와 선택 근거 확인
3. 예외/엣지 케이스 → 입력 검증, 연결 테스트, 저장 예외 대응
4. 프로토타입      → Credential 필드 명세 및 실제 화면 확인
```
